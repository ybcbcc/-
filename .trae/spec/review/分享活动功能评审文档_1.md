● 评审结论：当前不建议上线。
  主要风险集中在“分享奖励可被绕过刷积分”和“积分并发更新不一致”，属于上线阻断级问题。

  风险等级: P0 阻断
  位置: backend/service/share_service.go:162、backend/service/share_service.go:195
  问题: 分享奖励接口不强制要求有效 shareCode，且不存在的 shareCode 也会继续发奖
  失败场景 / 影响: 任意登录用户只要知道一个活动 ID 和其他用户 ID，就可以请求 /api/lottery/share/reward 给对方加 10 分；也可以伪造不存在的 shareCode 获得奖励。会导致积分被刷、奖励记录不可追溯
  修改建议: shareCode 必须非空；必须查到 LotteryShareRecord；且必须匹配 lotteryId + shareUserId，否则拒绝发奖
  ────────────────────────────────────────
  风险等级: P0 阻断
  位置: backend/service/share_service.go:187、backend/service/share_service.go:256、backend/service/share_service.go:259
  问题: 并发奖励会丢积分
  失败场景 / 影响: 两个不同好友几乎同时打开同一分享链接时，两个请求都读取到旧积分，例如 100，各自加 10 后都保存 110，最终应该是 120 但只变成 110；奖励记录有两条，用户积分只加了一次
  修改建议: 在事务内使用原子更新：UPDATE users SET integral = integral + 10，不要读取旧对象后 Save 整行
  ────────────────────────────────────────
  风险等级: P1 高
  位置: pages/lottery/detail.js:134、pages/lottery/detail.js:147、pages/lottery/detail.js:185
  问题: 前端先生成分享路径，分享成功后才异步创建分享记录
  失败场景 / 影响: 好友可能在 /api/lottery/share/create 完成前打开链接，或创建分享记录失败；此时链接里的 shareCode 没有后端记录。当前后端又允许不存在的 shareCode 发奖，会破坏可追溯性；后续如果修严校验，又会导致合法分享无法领奖  
  修改建议: 分享码应由后端提前生成并落库，再用于分享路径；不要把“创建分享记录”放在分享成功回调之后
  ────────────────────────────────────────
  风险等级: P1 高
  位置: pages/lottery/detail.json:5、pages/lottery/detail.js:139
  问题: 原生右上角分享菜单可绕过 shareReady，生成空 shareUserId 链接
  失败场景 / 影响: 页面刚打开时 currentUserId 还没加载完，用户用微信右上角菜单分享，path 会带 shareUserId=；好友打开后 pendingShareRewardCheck 为 false，不会触发奖励
  修改建议: onShareAppMessage 内也要校验 shareReady/currentUserId；或在分享参数未准备好时返回不带奖励口径的普通分享，并不要提示“可获得奖励”
  ────────────────────────────────────────
  风险等级: P2 中
  位置: backend/service/share_service.go:230、backend/service/share_service.go:252、backend/db/model/biz.go:130
  问题: 同一接收人并发重复打开时，唯一键冲突没有按幂等处理
  失败场景 / 影响: 两个重复请求都先查不到记录，随后一个插入成功，另一个命中唯一约束失败；失败请求返回 "Failed to reward share"，而不是预期的 already_rewarded
  修改建议: 捕获唯一键冲突并转换成 already_rewarded；或使用数据库 upsert / insert-ignore 风格实现幂等
