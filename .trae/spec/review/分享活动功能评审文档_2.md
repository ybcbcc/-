● 复审结论：当前仍不建议直接上线。
  本轮复审确认上次的 P0 阻断问题已基本修复：后端已要求有效 shareCode，奖励发放已改为事务内原子递增，并发重复奖励也改为幂等处理。
  但当前仍存在分享链路体验与数据准确性风险，主要集中在“原生分享菜单可生成无奖励链接”和“页面浏览即创建分享记录”。建议修复 P1 风险后再进入正式上线。

  风险等级: P1 高
  位置: pages/lottery/detail.js:137、pages/lottery/detail.json:5
  问题: 原生右上角分享菜单仍可在分享票据准备完成前生成不带奖励参数的分享链接
  失败场景 / 影响: 用户刚进入活动详情页时，shareReady/currentUserId/shareCode 还未准备好，但 detail.json 已启用原生分享菜单；此时从微信右上角菜单分享会返回 /pages/lottery/detail?id=...，不包含 shareUserId/shareCode。好友打开后 pendingShareRewardCheck 不会触发，分享人无法获得奖励，用户会认为分享成功但奖励链路实际失效。
  修改建议: onShareAppMessage 在分享参数未准备好时不要返回“正常可分享但无奖励”的路径；可选择禁用原生菜单入口、等待分享票据准备完成后再允许分享，或明确把未准备好的分享降级为普通分享且不作为奖励分享入口。

  位置: pages/lottery/detail.js:169、pages/lottery/detail.js:176、pages/lottery/detail.js:206、backend/service/share_service.go:88、backend/service/share_service.go:110
  问题: 分享记录在页面加载时创建，而不是用户实际发起分享时创建
  失败场景 / 影响: 任意已登录用户只要打开活动详情页，prepareShareEnv() 就会调用 ensureShareTicket(true)，随后请求 /api/lottery/share/create，后端会生成 shareCode 并插入 LotteryShareRecord。即使用户没有点击分享，也会产生分享发起记录，导致分享审计和统计数据虚高，分享记录表随普通浏览增长。
  修改建议: 将分享记录创建时机调整到用户实际触发分享前；可在点击自定义分享按钮时预创建 shareCode，或区分“预生成分享票据”和“实际分享发起记录”，避免把页面浏览计入分享发起。

  风险等级: P2 中
  位置: pages/lottery/detail.js:239、pages/lottery/detail.js:275、utils/request.js:28、app.js:26
  问题: 登录过期重试只覆盖 /api/user/info，没有覆盖后续分享创建和回流奖励接口
  失败场景 / 影响: loadCurrentUserIdWithRetry() 能在 /api/user/info 返回 401 时刷新 token，但如果 /api/lottery/share/create 或 /api/lottery/share/reward 自身返回 401，当前逻辑只记录日志并清空或保留状态，不会立即重新登录并重试。分享票据创建可能失败，或接收人打开有效分享链接时奖励未及时发放，只能依赖后续 onShow 或用户重新进入页面恢复。
  修改建议: 对分享相关接口统一增加 401 后刷新 token 并重试一次的逻辑；或在 request 层实现可控的认证重试，确保 create/reward 这类关键接口不会因为短暂 token 过期而静默失败。
