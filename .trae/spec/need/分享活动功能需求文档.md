# 分享活动功能需求文档

## 1. 文档信息

- 功能名称：活动分享与分享积分
- 适用项目：`wxxcx` 微信小程序
- 文档目的：基于当前项目结构，补充“活动详情页分享活动给微信好友并奖励积分”的产品需求，供后续设计、开发、测试对齐使用。

## 2. 项目现状理解

### 2.1 当前项目概况

当前项目为微信原生小程序前端配合 Go 后端服务的业务系统，核心能力包括：

- 首页活动/抽奖列表展示
- 抽奖活动发布、详情查看、报名参与
- 用户积分体系
- 用户个人中心、发布记录、参与记录、会员信息
- 后台审核与活动公告能力

### 2.2 与本需求直接相关的现状

1. 当前“活动详情页”实际对应抽奖活动详情页，位于：
   - `pages/lottery/detail.js`
   - `pages/lottery/detail.wxml`

2. 当前详情页底部只有一个主按钮：
   - 报名未开始 / 活动已结束 / 人数已满 / 立即报名
   - 暂无独立的分享入口

3. 当前项目已存在积分体系：
   - 用户积分字段：`backend/db/model/biz.go` 中的 `User.Integral`
   - 报名活动时会扣减积分：`backend/service/lottery_service.go`
   - 个人中心可展示当前积分：`pages/profile/profile.js`

4. 当前项目尚未发现分享能力的现有实现：
   - 未配置活动详情页分享按钮
   - 未实现 `onShareAppMessage`
   - 未实现分享记录与分享奖励发放逻辑

## 3. 需求背景

为了提升活动传播效率和用户活跃度，需要在活动详情页增加“分享活动”能力。用户可将当前活动分享给微信好友，并在满足规则时获得积分奖励，从而形成“活动传播 + 用户激励”的闭环。

## 4. 需求目标

### 4.1 核心目标

- 支持用户从活动详情页将当前活动分享给微信好友
- 分享入口放置在屏幕右上角的合适位置，保证易发现且不影响主内容浏览
- 分享内容需携带当前活动信息，好友点击后可进入对应活动详情页
- 同一活动分享给不同好友时，可分别获得奖励
- 每个符合条件的不同好友为分享人带来 10 积分奖励
- 分享结束后前端给出 toast 提示
- 平台具备分享记录、回流识别与奖励发放能力

### 4.2 非目标

本次需求暂不包含以下内容：

- 分享到朋友圈
- 海报生成
- 分享排行榜
- 多档积分奖励策略
- 独立积分明细页面

## 5. 角色与使用场景

### 5.1 角色

- 分享人：在活动详情页发起分享并获取奖励的用户
- 接收人：收到活动分享卡片并点击进入活动详情页的用户
- 系统：生成分享内容、识别回流用户、记录分享行为、发放积分

### 5.2 典型场景

1. 分享人进入某个活动详情页。
2. 分享人点击右上角“分享活动”按钮。
3. 系统应保证用户单次点击即可拉起微信分享面板，分享当前活动给微信好友。
4. 分享结束后，前端弹出 toast 提示。
5. 不同好友分别点击分享卡片进入活动详情页。
6. 系统识别这是不同的接收人，并按规则为分享人分别发放积分。

## 6. 功能需求

### 6.1 页面需求

#### 6.1.1 活动详情页增加分享入口

在活动详情页屏幕右上角增加固定分享按钮。

页面要求：

- 按钮位于右上角安全区域内
- 按钮悬浮展示，不遮挡标题等核心信息
- 页面滚动时按钮保持在右上角可点击区域
- 按钮文案或图标语义清晰，可直接触发微信分享

建议样式：

- 右上角悬浮按钮或胶囊式按钮
- 采用分享图标 + “分享”活动文案
- 与页面主视觉保持统一风格

#### 6.1.2 分享文案与内容

分享活动时，需要携带当前活动的核心信息，建议包含：

- 分享标题：活动标题
- 分享路径：活动详情页路径 + 当前活动 ID + 分享人 ID + 分享唯一标识
- 分享封面：当前活动主图
- 分享描述：可复用活动简介或奖品信息进行简化展示

示例路径：

```text
/pages/lottery/detail?id={lotteryId}&shareUserId={userId}&shareCode={shareCode}&shareSource=top_button
```

### 6.2 分享结束提示需求

用户完成一次分享操作后，前端需要给出 toast 提示。

提示要求：

- 分享成功后提示：`分享成功`
- 若需要引导奖励口径，可提示：`分享成功，好友打开后可获得奖励`
- 用户取消分享或分享失败时，不弹出成功 toast

### 6.3 积分奖励需求

#### 6.3.1 奖励规则

当分享人将同一活动分享给不同好友，且不同好友分别通过分享卡片进入活动详情页后，系统向分享人发放奖励。

基础奖励值：

- 每个符合条件的不同好友奖励：10 积分

#### 6.3.2 奖励发放原则

为防止重复刷分，需求约定如下：

- 以“同一活动 + 同一分享人 + 不同接收人”为奖励判定维度
- 同一接收人对同一活动仅可为同一分享人贡献 1 次奖励
- 同一活动分享给多个不同接收人时，可累计获得多次奖励
- 同一接收人反复打开同一分享链接，不重复加分
- 奖励发放必须可追踪，可查询到对应活动、分享人、接收人、奖励分值、奖励时间

#### 6.3.3 奖励提示

奖励发放成功后，系统需要给出明确提示或可查询反馈，例如：

- `积分 +10`
- `该好友已完成奖励，不重复发放`

### 6.4 分享记录需求

系统需新增分享记录能力，用于支撑回流识别、去重、追溯和后续统计。

建议记录两类数据：

1. 分享票据/分享发起记录
   - 记录 ID
   - 活动 ID
   - 分享人用户 ID
   - 分享唯一标识 `shareCode`
   - 分享渠道
   - 分享来源页面
   - 分享状态：`prepared` / `shared`
   - 实际分享确认时间
   - 创建时间
   - 更新时间

2. 分享回流奖励记录
   - 活动 ID
   - 分享人用户 ID
   - 接收人用户 ID
   - 对应分享唯一标识
   - 奖励积分值
   - 是否已发放奖励
   - 发放时间
   - 创建时间
   - 更新时间

### 6.5 分享回流需求

当好友通过分享卡片进入小程序时：

- 系统应识别其来源于某次活动分享
- 页面应直接打开对应活动详情页
- 系统应判断当前打开者是否为新的接收人
- 若为新的有效接收人，则为分享人增加 10 积分

## 7. 业务规则

### 7.1 活动范围

- 仅对活动详情页的分享生效
- 当前阶段仅覆盖抽奖活动详情页
- 未登录用户不能获得分享积分

### 7.2 奖励发放规则

- 奖励对象为当前发起分享的登录用户
- 奖励积分固定为 10 分
- 奖励依据为“不同接收人”而非“点击分享次数”
- 同一接收人对同一活动只计算一次奖励
- 分享人与接收人为同一用户时，不发放奖励
- 分享奖励与报名扣减积分相互独立
- 分享奖励发放后，应同步更新用户积分总额

### 7.3 异常规则

- 用户身份失效时，不允许发放奖励
- 活动不存在时，不允许生成分享奖励记录
- 奖励发放失败时，不应出现“前端提示已到账但后台未加分”的状态
- 同一请求重复提交时，后台需保证幂等

## 8. 产品交互流程

### 8.1 正常流程

1. 用户进入活动详情页。
2. 用户点击右上角分享按钮。
3. 小程序直接触发微信分享面板。
4. 用户完成分享后，系统将本次分享票据确认为真实分享，前端 toast 提示“分享成功”。
5. 好友点击分享卡片进入活动详情页。
6. 系统识别活动 ID、分享人 ID、分享唯一标识以及当前打开用户。
7. 若当前打开用户为新的有效接收人，则新增回流奖励记录并给分享人积分加 10。

### 8.2 重复接收人流程

1. 分享人已因某个接收人获得过本活动奖励。
2. 同一接收人再次点击相同活动的分享卡片。
3. 系统识别该接收人已为该分享人贡献过奖励。
4. 本次允许正常进入活动详情页。
5. 系统不重复加分。

## 10. 接口需求建议

以下为需求层建议，并非最终接口定稿。

### 10.1 创建分享记录

接口用途：

- 预生成可用于分享路径的分享唯一标识
- 创建 `prepared` 状态的分享票据
- 不直接记为最终分享成功统计

建议接口：

```text
POST /api/lottery/share/create
```

建议请求参数：

```json
{
  "lotteryId": "活动ID",
  "shareSource": "top_button",
  "shareChannel": "wechat_friend"
}
```

建议返回参数：

```json
{
  "shareCode": "分享唯一标识",
  "shareStatus": "prepared",
  "message": "share_prepared"
}
```

### 10.2 确认真实分享

接口用途：

- 在微信分享成功后，将对应分享票据确认为真实分享
- 将分享状态从 `prepared` 更新为 `shared`

建议接口：

```text
POST /api/lottery/share/confirm
```

建议请求参数：

```json
{
  "lotteryId": "活动ID",
  "shareCode": "分享唯一标识",
  "shareSource": "top_button"
}
```

### 10.3 分享回流奖励

接口用途：

- 识别打开分享链接的接收人
- 校验该接收人是否已贡献过奖励
- 为分享人发放积分

建议接口：

```text
POST /api/lottery/share/reward
```

建议请求参数：

```json
{
  "lotteryId": "活动ID",
  "shareUserId": "分享人ID",
  "shareCode": "分享唯一标识"
}
```

建议返回参数：

```json
{
  "rewarded": true,
  "points": 10,
  "message": "积分 +10",
  "currentIntegral": 1010
}
```

### 10.4 分享回流落地

打开分享链接时，页面需能接收：

- `id`：活动 ID
- `shareUserId`：分享人 ID
- `shareCode`：分享唯一标识
- `shareSource`：分享来源

## 11. 数据需求建议

建议新增两张表。

### 11.1 分享发起表

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| lottery_id | string | 活动 ID |
| share_user_id | string | 分享人用户 ID |
| share_code | string | 分享唯一标识 |
| share_channel | string | 分享渠道 |
| share_source | string | 来源按钮 |
| share_status | string | 分享状态，`prepared` 或 `shared` |
| shared_at | datetime | 实际分享确认时间 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

建议唯一约束：

- `share_code` 唯一

### 11.2 分享奖励表

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| id | bigint | 主键 |
| lottery_id | string | 活动 ID |
| share_user_id | string | 分享人用户 ID |
| receiver_user_id | string | 接收人用户 ID |
| share_code | string | 分享唯一标识 |
| reward_points | int | 奖励积分，默认 10 |
| rewarded | bool | 是否已发放奖励 |
| rewarded_at | datetime | 奖励发放时间 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

建议唯一约束：

- `lottery_id + share_user_id + receiver_user_id` 唯一

该约束可确保“同一活动中，同一接收人只为同一分享人贡献一次奖励”。

## 13. 风险与待确认项

### 13.1 为什么能识别回流用户

需要明确区分两个阶段：

- 分享发起阶段：系统不知道用户最终把消息发送给了微信中的哪位好友
- 分享回流阶段：系统可以识别“是谁打开了这条分享链接”

原因如下：

- 微信小程序在分享时，不会把“接收好友身份”直接返回给业务系统
- 但当某个用户点击分享卡片进入小程序时，这个用户会以“当前访问者”的身份打开页面
- 页面会拿到分享参数，例如活动 ID、分享人 ID、分享唯一标识
- 后端再结合当前访问者登录态，就可以识别出“当前打开链接的人是谁”

因此，系统不能判断：

- 用户在发送瞬间到底选中了哪位好友

但系统可以判断：

- 当前是谁通过这条分享链接进入了活动详情页

本需求中的“不同的人”实际指：

- 不同的登录回流用户

### 13.2 “不同的人”判定口径

本需求中的“不同的人”建议定义为：

- 不同的登录接收用户

即：

- 不是按“分享给不同聊天窗口”判定
- 不是按“点击分享按钮次数”判定
- 而是按“不同接收人通过分享链接进入小程序”判定

### 13.3 奖励到账时机

本次建议采用：

- 分享结束后立即 toast 提示“分享成功”
- 奖励在不同接收人打开分享链接后发放

这样可以避免“仅点击分享按钮就刷分”的问题。

### 13.4 分享统计真实性

为避免“用户只点击按钮但未真正分享”导致统计虚高，需求补充约束如下：

- 分享票据预生成与真实分享发起必须区分
- 页面可预生成 `prepared` 状态票据，以保证单次点击即可分享
- 只有分享成功后，票据才可更新为 `shared`
- 分享统计、审计与后续分析应以 `shared` 状态为准

## 14. 评审吸收记录

### 14.1 第一轮评审吸收摘要

- 强化了 `shareCode` 必填且必须命中真实分享记录的要求
- 明确奖励发放需具备事务与幂等保护
- 明确分享链接不能绕过准备态直接生成无奖励参数

### 14.2 第二轮评审吸收摘要

- 增加“原生右上角分享菜单不作为奖励分享入口”的约束
- 增加“分享相关接口需支持 401 后重新登录重试”的要求
- 删除“页面浏览即视为分享发起”的需求口径

### 14.3 第三轮评审吸收摘要

- 明确“单次点击直接拉起分享面板”是必须满足的交互要求
- 明确分享票据与真实分享记录必须分状态管理，避免统计失真
- 将分享发起记录补充为 `prepared` / `shared` 两阶段状态

### 13.4 防刷策略待确认

建议确认以下规则是否纳入首版：

- 是否限制每日最多奖励次数
- 是否限制单个活动的最高可获奖励人数
- 是否需要排除未登录回流用户

## 14. 实施建议

为保证首版需求可控，建议按以下优先级推进：

### 14.1 第一阶段

- 详情页右上角新增分享按钮
- 可将当前活动分享给微信好友
- 分享结束后前端 toast 提示
- 新增分享发起记录

### 14.2 第二阶段

- 分享回流统计

基于当前项目现状，本需求适合落在“抽奖活动详情页”中实现，依托现有用户积分体系扩展“右上角分享入口 + 分享回流识别 + 多接收人奖励”能力。首版建议采用“分享成功后 toast 提示、不同接收人回流后给分享人奖励 10 积分”的方案。
