# 活动时间时区修复技术文档

## 1. 文档目标

本文档用于说明“活动时间时区修复”在当前项目中的技术落地方案，覆盖前后端改造点、时间口径约定、接口设计、关键时序、兼容性方案、历史数据修复方案与测试重点。

## 2. 设计原则

- 活动模块统一以北京时间作为业务时间口径
- 前后端之间只允许一处做时区转换，避免重复偏移
- 前端不再提交 UTC ISO 时间，也不再手动减 8 小时显示
- 后端显式按北京时间解析活动开始时间
- 活动有效期判断使用明确的北京时间基准，不依赖模糊环境时区
- 历史脏数据修复与代码改造解耦，先保证新数据正确，再修复旧数据

## 3. 当前问题分析

### 3.1 当前错误链路

当前链路可概括为：

1. 管理员在活动发布页选择北京时间。
2. 前端通过 `new Date(...).toISOString()` 将本地时间转成 UTC 字符串。
3. 后端将该 UTC 时间直接反序列化并保存到 `activity.start_time`。
4. 前端编辑页与列表页又对返回时间执行减 8 小时处理。
5. 后端活动有效期查询使用数据库 `NOW()` 与 `start_time` 比较，环境时区不清晰。

### 3.2 结果

- 数据库存储语义与运营配置语义不一致
- 页面展示时间与运营配置不一致
- 活动状态判断提前 8 小时
- 编辑后再次保存可能出现二次偏移

## 4. 总体方案

### 4.1 最终口径

本次采用以下统一方案：

- 前端提交格式：`YYYY-MM-DD HH:mm:ss`
- 后端接收格式：字符串
- 后端解析方式：`time.ParseInLocation(..., bj)`
- 数据库存储语义：北京时间对应的 `datetime`
- 前端展示方式：直接按北京时间格式化，不再做 `-8h`
- 活动状态计算：前后端均按北京时间进行

### 4.2 为什么不继续沿用当前 UTC 方案

理论上也可以采用“统一存 UTC、统一返回时区”的方案，但当前项目已有活动表 `datetime` 字段、页面展示逻辑和广告有效期判断都围绕本地业务时间展开。首版修复以最小改动、最低回归风险为目标，优先选择“前后端统一按北京时间字符串处理”的方案。

## 5. 前端改造方案

### 5.1 活动发布页

改造文件：

- `pages/activity/publish/publish.js`

改造点：

1. 提交时不再执行 `toISOString()`
2. 用户选择的 `startDate` 与 `startTime` 直接拼接为 `YYYY-MM-DD HH:mm:ss`
3. 编辑页回显时不再执行减 8 小时逻辑
4. 补充统一的北京时间解析与格式化辅助函数

建议实现：

- 新增 `formatDateTimeFields(value)`，将后端返回值转成页面需要的日期与时间字段
- 新增 `buildStartTimeText(startDate, startTime)`，负责生成提交字符串

### 5.2 活动列表页

改造文件：

- `pages/activity/list/list.js`

改造点：

1. 保留“将字符串解析为北京时间毫秒值”的工具能力
2. 删除展示逻辑中的 `- 8 * 3600 * 1000`
3. 删除筛选逻辑中的 `adjStart`
4. 统一以北京时间毫秒值直接计算开始、结束和当前状态

### 5.3 前端兼容策略

- 若后端返回 `YYYY-MM-DD HH:mm:ss`，前端按 `+08:00` 解析
- 若后端暂时返回带 `T` 与 `Z` 的旧格式，前端仍兼容解析
- 页面只保留“兼容旧返回值”的解析能力，不再主动做时间偏移

## 6. 后端改造方案

### 6.1 活动请求结构

改造文件：

- `backend/service/activity_service.go`

当前问题：

- `ActivityCreateRequest.StartTime` 与 `ActivityUpdateRequest.StartTime` 使用 `time.Time`
- `json.Decoder` 直接解析时没有明确“按北京时间解析”的业务语义

改造方案：

- 将 `StartTime` 字段改为 `string`
- 新增统一解析函数，例如：

```go
func parseActivityStartTime(value string) (time.Time, error)
```

解析规则：

- 输入格式：`2006-01-02 15:04:05`
- 使用 `time.FixedZone("CST", 8*3600)` 解析
- 解析失败返回明确错误

### 6.2 活动创建与更新

改造文件：

- `backend/service/activity_service.go`

改造点：

1. 创建活动时调用统一解析函数
2. 更新活动时调用统一解析函数
3. 对空时间或非法时间返回明确错误
4. `CreatedAt`、`UpdatedAt` 保持现有逻辑

### 6.3 活动详情返回

活动详情接口保持返回 `activity` 结构体，但要保证其 `StartTime` 在 JSON 序列化后不会要求前端再额外减 8 小时。

首版可接受两种技术落地方式：

- 方式 A：返回默认 JSON 时间格式，但前端只做解析不做减 8 小时
- 方式 B：在后端构造返回 DTO，显式输出 `YYYY-MM-DD HH:mm:ss`

结合当前项目改造成本，首版建议采用方式 A，并由前端兼容解析。

### 6.4 活动有效期查询

改造文件：

- `backend/db/dao/biz_impl.go`

当前问题：

- 使用 SQL `NOW()` 作为当前时间基准
- 当数据库、容器、进程时区不一致时，判断结果存在偏差风险

改造方案：

- 在 Go 侧生成北京时间当前时刻
- 通过参数传入 SQL 参与比较

建议实现方式：

```go
bj := time.FixedZone("CST", 8*3600)
nowText := time.Now().In(bj).Format("2006-01-02 15:04:05")
Where("start_time <= ? AND DATE_ADD(start_time, INTERVAL duration_minutes MINUTE) >= ?", nowText, nowText)
```

适用范围：

- `GetActiveActivities`
- `GetLatestActiveActivity`

### 6.5 数据库连接时区

改造文件：

- `backend/db/init.go`

当前问题：

- DSN 使用 `loc=Local`
- 当服务部署环境不是东八区时，`Local` 含义不稳定

改造方案：

- 进程启动时通过 `time.FixedZone("CST", 8*3600)` 将 `time.Local` 固定为东八区
- DSN 继续使用 `loc=Local`
- 数据库连接建立后执行 `SET time_zone = '+08:00'`

说明：

- 该方案不依赖容器镜像内置 `Asia/Shanghai` 时区数据，避免在精简镜像中出现 `unknown time zone` 启动失败
- `loc=Local` 在本方案下等价于东八区
- 即使 SQL 侧已传入北京时间，该改造仍作为环境兜底

## 7. 数据模型与接口影响

### 7.1 数据模型

本次不新增表结构，不新增字段。

活动表保持：

- `start_time`
- `duration_minutes`
- `appearance_count`

### 7.2 接口影响

受影响接口：

- `POST /api/activity/create`
- `POST /api/activity/update`
- `GET /api/activity/detail`
- `GET /api/activity/list`
- `GET /api/activity/ad/latest`

兼容性说明：

- 创建与更新接口的 `startTime` 从“ISO 时间字符串或自动反序列化时间对象”收敛为“北京时间字符串”
- 需要同步调整小程序请求参数格式

## 8. 关键时序

### 8.1 创建活动

1. 管理员在前端选择日期与时间。
2. 前端拼接 `YYYY-MM-DD HH:mm:ss`。
3. 前端调用 `/api/activity/create`。
4. 后端按北京时间解析字符串。
5. 后端将解析后的时间写入 `activity.start_time`。
6. 后端返回活动数据。
7. 前端后续读取该活动时直接按北京时间显示。

### 8.2 编辑活动

1. 前端调用活动详情接口获取 `startTime`。
2. 前端将返回值解析为北京时间，不再减 8 小时。
3. 页面拆分出 `startDate` 和 `startTime` 进行回显。
4. 用户保存时再次提交北京时间字符串。

### 8.3 活动有效性判断

1. 后端获取当前北京时间。
2. DAO 使用北京时间与 `start_time`、`duration_minutes` 计算活动时间窗。
3. 命中窗口内的活动视为进行中。
4. 活动列表与活动广告接口共用同一时间判断口径。

## 9. 历史数据修复方案

### 9.1 修复原则

- 只修复已确认受影响的数据
- 先查询、后更新
- 优先按活动 ID 执行，降低误修风险

### 9.2 单条修复 SQL

```sql
SELECT id, name, start_time, DATE_ADD(start_time, INTERVAL 8 HOUR) AS fixed_start_time
FROM activity
WHERE id = ?;

UPDATE activity
SET start_time = DATE_ADD(start_time, INTERVAL 8 HOUR),
    updated_at = NOW()
WHERE id = ?;
```

### 9.3 批量修复策略

- 首选按活动 ID 列表批量修复
- 次选按受影响版本创建时间范围筛选
- 不允许直接对全表统一加 8 小时

### 9.4 修复校验

- 修复前记录原始 `start_time`
- 修复后核对页面回显
- 修复后核对活动状态是否恢复正常

## 10. 联调与测试重点

### 10.1 前端联调

- 创建活动后检查请求体中的 `startTime` 是否为北京时间字符串
- 编辑活动回显是否与创建值一致
- 列表中展示时间是否不再提前 8 小时

### 10.2 后端联调

- 创建活动时非法时间格式是否报错
- 更新活动时非法时间格式是否报错
- 有效活动查询与广告查询是否命中一致

### 10.3 场景测试

- `12:00` 开始、持续 60 分钟的活动，在 `11:59 / 12:30 / 13:01` 的状态是否正确
- 同一活动保存两次后时间是否保持不变
- 老活动修复后是否与运营原始配置一致

## 11. 风险与兼容性

### 11.1 风险点

- 线上可能已有部分活动被人工修正，批量修复需谨慎
- 若还有其他页面复用了旧时间偏移逻辑，可能存在遗漏
- JSON 默认时间序列化格式若与前端假设不一致，需要补充 DTO 返回层

### 11.2 兼容方案

- 前端解析先兼容旧格式与新格式
- 后端请求解析立即收敛到新格式
- 数据修复独立执行，避免与代码上线耦合

## 12. 实施步骤建议

1. 先修改前端活动发布页与列表页时间处理逻辑。
2. 再修改后端活动创建、更新、有效期判断逻辑。
3. 调整数据库连接时区配置。
4. 进行本地联调与状态回归验证。
5. 最后执行线上脏数据修复。

## 13. 结论

本方案通过“前端提交北京时间字符串、后端按北京时间显式解析、前后端取消重复减 8 小时、DAO 使用明确北京时间判断”的组合改造，可在不扩表的前提下修复活动时间提前 8 小时的问题，并为历史错误数据提供可控修复路径。
