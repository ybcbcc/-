# 分享活动功能技术文档

## 1. 文档目标

本文档用于说明“活动详情页分享活动并奖励积分”功能在当前项目中的技术落地方案，覆盖前端改造、后端接口、数据模型、流程时序、幂等与风控要点。

## 2. 现有项目基础

### 2.1 前端现状

- 活动详情页当前位于 `pages/lottery/detail.js` 和 `pages/lottery/detail.wxml`
- 详情页已具备活动数据拉取、报名按钮状态控制、报名接口调用等能力
- 当前页面尚未实现 `onShareAppMessage`
- 当前页面尚无右上角分享按钮

### 2.2 后端现状

- 活动详情接口：`/api/lottery/detail`
- 活动报名与结果接口：`/api/lottery/draw`
- 用户积分字段：`backend/db/model/biz.go` 中的 `User.Integral`
- 当前已有积分扣减与积分增加逻辑，但没有分享奖励逻辑

### 2.3 核心约束

由于微信小程序不会直接返回“用户分享给了哪个好友”，因此“分享给不同的人可获得奖励”不能以点击分享按钮次数作为依据，而应以“不同接收人通过分享链接回流进入小程序”作为技术判定口径。

### 2.4 为什么能识别回流用户

技术上需要区分两个动作：

1. 用户发起分享
2. 其他用户打开分享链接

在第 1 步中，微信不会把“分享消息最终发送给了谁”直接回传给小程序，因此系统无法在分享发生当下拿到接收好友身份。

在第 2 步中，若某个用户点击分享卡片并进入小程序，则系统可以同时拿到两类信息：

- 分享链路参数
  - `lotteryId`
  - `shareUserId`
  - `shareCode`
- 当前访问者身份
  - 当前打开页面的登录用户

因此后端能够判断：

- 这条分享是谁发起的
- 当前是谁打开了这条分享链接

但后端不能判断：

- 用户在微信聊天窗口里最初发送给的是谁

所以本方案中的“分享给不同的人奖励”在技术实现上等价于：

- 不同的回流登录用户分别打开分享链接时，分别触发奖励判定

## 3. 总体实现方案

### 3.1 设计原则

- 分享入口放在活动详情页右上角固定区域
- 分享成功后立即给用户 toast 提示
- 奖励不在点击分享时立即发放
- 奖励在不同接收人打开分享链接后发放
- 同一活动中，同一接收人只为同一分享人贡献一次奖励
- 所有奖励逻辑由后端统一判定，前端不做最终奖励结果裁决

### 3.2 功能拆分

整体拆分为五个步骤：

1. 页面提前为当前登录用户预生成 `prepared` 状态的分享票据
2. 分享人单次点击右上角分享按钮并直接拉起分享面板
3. `onShareAppMessage` 使用已准备好的 `shareCode` 生成分享路径
4. 用户完成微信分享后，前端调用确认接口将票据更新为 `shared`
5. 接收人通过分享链接进入详情页时，后端校验是否为新的有效接收人，若是则给分享人加 10 积分

## 4. 前端实现方案

### 4.1 页面改造点

改造文件：

- `pages/lottery/detail.wxml`
- `pages/lottery/detail.wxss`
- `pages/lottery/detail.js`

### 4.2 UI 方案

在详情页新增右上角悬浮分享按钮。

建议实现方式：

- 按钮定位在顶部安全区右侧
- 使用 `position: fixed` 或基于页面头图容器的绝对定位
- 采用 `open-type="share"` 触发分享
- 使用图标加短文案，例如“分享”

建议样式要求：

- 不遮挡系统胶囊按钮区域
- 不遮挡活动标题与 banner 主要视觉
- 在 Android 和 iOS 上均能正常点击

### 4.3 页面数据新增

建议在 `pages/lottery/detail.js` 的 `data` 中新增：

```js
data: {
  lotteryId: null,
  lotteryInfo: {},
  shareCode: '',
  shareSource: 'top_button',
  pendingShareRewardCheck: false
}
```

字段说明：

- `shareCode`：当前页面最近一次创建的分享唯一标识
- `shareSource`：分享入口来源，固定为右上角按钮
- `pendingShareRewardCheck`：用于标识当前页面是否需要触发分享回流奖励校验

### 4.4 页面生命周期处理

#### 4.4.1 onLoad

在 `onLoad(options)` 中增加对分享参数的识别：

- `id`
- `shareUserId`
- `shareCode`
- `shareSource`

若存在 `shareUserId` 和 `shareCode`，说明当前页面由分享回流进入，应记录到页面状态，用于后续触发奖励校验。

#### 4.4.2 onShow

建议在 `onShow` 中判断是否需要执行分享回流奖励校验，原因：

- 小程序分享进入页面时，登录态、用户信息可能尚未完全可用
- 可在页面展示后再调用奖励接口，减少首次进入时机不稳定的问题

### 4.5 分享触发流程

建议实现步骤如下：

1. 页面加载后先获取当前用户身份
2. 前端调用 `/api/lottery/share/create` 预生成 `prepared` 状态的 `shareCode`
3. 用户点击右上角分享按钮
4. 页面触发 `onShareAppMessage`
5. 返回分享配置对象：
   - `title`
   - `path`
   - `imageUrl`
6. 用户完成分享后触发成功回调
7. 前端调用 `/api/lottery/share/confirm`，将票据标记为 `shared`
8. 前端弹出 toast：
   - `分享成功`
   - 或 `分享成功，好友打开后可获得奖励`

### 4.6 分享配置示例

建议路径格式：

```text
/pages/lottery/detail?id={lotteryId}&shareUserId={userId}&shareCode={shareCode}&shareSource=top_button
```

建议分享返回对象包含：

```js
{
  title: lotteryInfo.title,
  path,
  imageUrl: lotteryInfo.imageUrl
}
```

### 4.7 分享成功 toast

分享成功后的 toast 建议：

- 文案：`分享成功，好友打开后可获得奖励`
- 图标：`none`
- 时长：1500ms 到 2000ms

取消分享或失败建议：

- 不提示成功
- 可根据体验决定是否静默处理

### 4.8 分享回流奖励校验

当页面通过分享参数打开时：

1. 校验当前登录用户是否存在
2. 校验 `shareUserId` 与当前用户是否相同
3. 若不是本人打开，则调用后端奖励接口
4. 接口返回 `rewarded = true` 时，可静默处理或在适当时机展示奖励状态

注意：

- 当前页面是接收人页面，奖励对象是分享人
- 因此接收人页面不应显示“你获得了积分”，除非产品需要显式提示
- 如果需要给分享人反馈奖励到账，可考虑后续增加消息通知或积分刷新机制

## 5. 后端实现方案

### 5.1 服务归属建议

建议新增分享相关服务逻辑，方式有两种：

- 方案 A：在 `lottery_service.go` 中直接扩展分享接口
- 方案 B：新增 `share_service.go`，将分享逻辑独立维护

建议采用方案 B，原因：

- 便于与报名、开奖逻辑解耦
- 分享奖励会逐步扩展统计、风控、通知等能力
- 独立服务更清晰

### 5.2 接口列表

建议新增三个接口：

#### 5.2.1 预生成分享票据

```text
POST /api/lottery/share/create
```

用途：

- 校验活动是否存在
- 校验分享人是否登录
- 创建 `prepared` 状态的分享票据
- 生成唯一 `shareCode`

#### 5.2.2 确认真实分享

```text
POST /api/lottery/share/confirm
```

用途：

- 校验当前分享票据是否属于当前登录分享人
- 在分享成功后将票据从 `prepared` 更新为 `shared`
- 为统计与审计提供真实分享依据

#### 5.2.3 处理分享回流奖励

```text
POST /api/lottery/share/reward
```

用途：

- 校验当前打开页面的用户身份
- 校验活动是否存在
- 校验分享人是否存在
- 校验当前打开用户是否为新的有效接收人
- 对符合条件的分享人加 10 积分
- 记录奖励结果

### 5.3 路由接入建议

在 `backend/main.go` 中新增路由，例如：

```go
http.HandleFunc("/api/lottery/share/create", service.LotteryShareCreateHandler)
http.HandleFunc("/api/lottery/share/reward", service.LotteryShareRewardHandler)
```

## 6. 数据模型设计

建议新增两张表。

### 6.1 分享发起表

表名建议：`lottery_share_record`

字段建议：

- `id`
- `lottery_id`
- `share_user_id`
- `share_code`
- `share_channel`
- `share_source`
- `share_status`
- `shared_at`
- `created_at`
- `updated_at`

索引建议：

- `share_code` 唯一索引
- `lottery_id + share_user_id` 普通索引

### 6.2 分享奖励表

表名建议：`lottery_share_reward`

字段建议：

- `id`
- `lottery_id`
- `share_user_id`
- `receiver_user_id`
- `share_code`
- `reward_points`
- `rewarded`
- `rewarded_at`
- `created_at`
- `updated_at`

索引建议：

- `lottery_id + share_user_id + receiver_user_id` 唯一索引
- `share_code` 普通索引
- `share_user_id` 普通索引

### 6.3 模型层建议

在 `backend/db/model/biz.go` 中新增两个结构体：

- `LotteryShareRecord`
- `LotteryShareReward`

建议同时在 DAO 层补充：

- 创建分享记录
- 根据 `shareCode` 查询分享记录
- 查询指定活动、分享人、接收人的奖励记录
- 创建奖励记录
- 更新奖励记录

## 7. 接口设计细节

### 7.1 创建分享记录接口

请求示例：

```json
{
  "lotteryId": "abc123",
  "shareSource": "top_button",
  "shareChannel": "wechat_friend"
}
```

响应示例：

```json
{
  "code": 0,
  "data": {
    "shareCode": "share_xxx",
    "shareSource": "top_button"
  }
}
```

处理逻辑：

1. 从 token 解析当前用户
2. 校验活动是否存在
3. 生成唯一 `shareCode`
4. 写入 `prepared` 状态的分享票据
5. 返回 `shareCode`

### 7.2 确认真实分享接口

请求示例：

```json
{
  "lotteryId": "abc123",
  "shareCode": "share_xxx",
  "shareSource": "top_button"
}
```

响应示例：

```json
{
  "code": 0,
  "data": {
    "shareCode": "share_xxx",
    "shareSource": "top_button",
    "shareStatus": "shared",
    "message": "share_confirmed"
  }
}
```

处理逻辑：

1. 从 token 解析当前分享人
2. 校验 `shareCode` 是否存在且属于当前用户和活动
3. 若当前状态为 `prepared`，则更新为 `shared`
4. 若已是 `shared`，按幂等直接返回成功

### 7.3 分享回流奖励接口

请求示例：

```json
{
  "lotteryId": "abc123",
  "shareUserId": "user_share_1",
  "shareCode": "share_xxx"
}
```

响应示例一：首次奖励

```json
{
  "code": 0,
  "data": {
    "rewarded": true,
    "points": 10,
    "message": "积分 +10"
  }
}
```

响应示例二：已奖励过

```json
{
  "code": 0,
  "data": {
    "rewarded": false,
    "points": 0,
    "message": "already_rewarded"
  }
}
```

处理逻辑：

1. 从 token 解析当前接收人用户
2. 校验活动是否存在
3. 校验 `shareCode` 是否存在且属于该活动和分享人
4. 接受 `prepared` 或 `shared` 状态的有效票据
5. 校验接收人和分享人不能相同
6. 查询 `lottery_id + share_user_id + receiver_user_id` 是否已有奖励记录
7. 若已有记录，直接返回 `rewarded = false`
8. 若无记录，开启事务
9. 写入奖励记录
10. 为分享人增加 10 积分
11. 回写分享票据状态为 `shared`
12. 提交事务并返回 `rewarded = true`

## 8. 事务与幂等设计

### 8.1 幂等目标

需要避免以下情况导致重复加分：

- 同一接收人重复打开同一个分享链接
- 同一接收人在网络抖动下重复请求奖励接口
- 接收人快速连续进入页面触发多次请求

### 8.2 实现方式

建议使用数据库唯一约束配合事务实现：

- 唯一键：`lottery_id + share_user_id + receiver_user_id`
- 插入奖励记录时若违反唯一约束，则说明奖励已发放过

### 8.3 事务边界

奖励接口中以下步骤需放在同一事务内：

1. 插入分享奖励记录
2. 更新分享人的积分

这样可以避免出现“记录已写入但积分未加”或“积分已加但记录未写入”的脏数据。

## 9. 风控与边界处理

### 9.1 必要规则

- 分享人和接收人相同，不奖励
- 活动不存在，不奖励
- 分享记录不存在，不奖励
- 分享记录与活动不匹配，不奖励
- 未登录接收人打开页面，不触发奖励

### 9.2 可扩展风控

后续可追加：

- 单活动奖励上限人数
- 单用户每日奖励上限
- 异常设备或异常 IP 识别
- 黑名单用户过滤

## 10. 前后端时序

### 10.1 发起分享时序

1. 分享人进入详情页
2. 页面预请求 `/api/lottery/share/create`
3. 后端返回 `prepared` 状态的 `shareCode`
4. 用户单次点击右上角分享按钮
5. 前端通过 `onShareAppMessage` 返回带 `shareCode` 的分享路径
6. 用户完成分享
7. 前端请求 `/api/lottery/share/confirm`
8. 后端将票据更新为 `shared`
9. 前端 toast 提示“分享成功，好友打开后可获得奖励”

### 10.2 奖励发放时序

1. 接收人点击分享卡片进入详情页
2. 页面读取 `shareUserId`、`shareCode`、`lotteryId`
3. 前端在合适时机请求 `/api/lottery/share/reward`
4. 后端校验并判定是否为新的有效接收人
5. 若是，则为分享人增加 10 积分并写奖励记录
6. 若票据仍为 `prepared`，则同步回写为 `shared`
7. 前端完成当前页面展示

### 10.3 身份识别说明

奖励判定依赖两类身份数据：

- 分享人身份：由分享链接中的 `shareUserId` 标识
- 回流用户身份：由当前访问页面的登录态标识

后端实际判定逻辑为：

```text
活动 = lotteryId
分享人 = shareUserId
回流用户 = currentUserId

如果 currentUserId != shareUserId
且 currentUserId 未对该活动贡献过奖励
则给 shareUserId 增加 10 积分
```

这意味着奖励依据不是“发给谁”，而是“谁最终点开了链接并进入了小程序”。

## 11. 开发步骤建议

1. 修改 `detail.wxml` 新增右上角分享按钮
2. 修改 `detail.wxss` 增加右上角浮层样式

4. 接入 `onShareAppMessage`
5. 增加分享成功 toast
6. 增加分享回流奖励调用

### 11.2 后端

1. 新增模型结构体
2. 新增 DAO 方法
3. 新增分享预生成、确认与奖励接口 handler
4. 新增事务化奖励逻辑
5. 在 `main.go` 注册路由

### 11.3 测试

1. 测试同一用户分享成功 toast 是否正常显示
2. 测试不同接收人回流时是否累计奖励
3. 测试同一接收人重复进入时是否不会重复加分
4. 测试自己打开自己的分享链接是否不会加分
5. 测试活动不存在、参数缺失、未登录场景

## 12. 验收关注点

- 右上角按钮位置是否兼容不同机型安全区
- 分享成功 toast 是否只在成功场景出现
- 回流奖励是否只按不同接收人计算
- 后端是否具备幂等与事务保护
- 个人中心积分是否能及时反映最新值

## 13. 结论

该需求在当前项目中可落地，推荐采用“右上角分享按钮 + 分享成功 toast + 不同接收人回流奖励”的技术方案。该方案与微信小程序平台能力匹配度较高，且能较好控制刷分风险，适合作为首版实现。

## 14. 评审吸收记录

### 14.1 第一轮评审吸收摘要

- 收紧 `shareCode` 校验，要求奖励接口必须命中真实分享记录
- 将积分增加改为事务内原子递增
- 将重复回流改为数据库幂等处理

### 14.2 第二轮评审吸收摘要

- 隐藏原生右上角分享菜单，避免生成无奖励链接
- 分享相关接口统一补充 401 后重新登录重试
- 删除“页面浏览即创建正式分享记录”的旧方案

### 14.3 第三轮评审吸收摘要

- 前端恢复单次点击直接拉起分享面板
- 后端新增 `prepared` / `shared` 双状态，区分预生成票据与真实分享
- 新增 `/api/lottery/share/confirm` 接口，用于在分享成功后确认真实分享
- 回流奖励接口兼容将有效 `prepared` 票据回写为 `shared`，降低确认失败带来的数据丢失风险
