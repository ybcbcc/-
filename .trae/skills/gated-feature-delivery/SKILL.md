# Gated Feature Delivery

用于执行“需求文档 -> 技术文档 -> 代码实现 -> 读取评审并迭代修正”的闸口式交付流程。

可配合以下子 Skill 使用：

- `gated-requirement-writer`
- `gated-tech-designer`
- `gated-review-absorber`

## 适用场景

当用户明确要求以下任一约束时，必须使用本 Skill：

- 先写需求文档，再写技术文档，再开始实现
- 每完成一个阶段都必须停下来，等待用户审核确认后再进入下一阶段
- 代码实现后需要结合评审文档继续迭代
- 评审意见需要同步沉淀回需求文档和技术文档
- 评审文件由别人提供，当前执行者只负责读取、理解、修正，不负责生成评审文档

典型触发语句：

- “先出需求文档，确认后再出技术文档”
- “技术文档确认后再改代码”
- “根据评审意见继续修改，但评审文档不是你写”
- “每个阶段做完都要停下来等我确认”

## 核心原则

### 1. 严格分阶段推进

流程固定为四个阶段，不允许跨阶段抢跑：

1. 需求阶段
2. 技术阶段
3. 实现阶段
4. 评审吸收与修正阶段

### 2. 每阶段结束必须暂停

每完成一个阶段后，必须：

- 明确说明当前阶段已完成
- 给出本阶段产物路径
- 提醒用户审核
- 停止执行，等待用户批准后再进入下一阶段

### 3. 文档持续回写

需求文档和技术文档不是一次性文件，而是持续维护的“活文档”。

每一轮都必须回写至少以下内容：

- 当前阶段新增结论
- 已实现范围
- 因评审而产生的变更
- 已修复问题和未修复问题
- 当前风险与待确认项

### 4. 评审文档只读

执行者不是评审方，职责仅包括：

- 读取用户提供的评审文档
- 判断评审意见是否合理
- 将合理意见转化为改动计划
- 把已吸收的评审意见和对应实现结果同步更新到需求文档、技术文档

职责不包括：

- 生成评审文档
- 伪造评审结论
- 代替评审方下结论

## 阶段要求

### 阶段 1：需求阶段

- 目标：产出或更新需求文档
- 最低要求：背景、目标、交互、规则、验收、风险
- 结束条件：文档已落盘并通知用户审核
- 闸口：未获批准不得进入技术阶段

### 阶段 2：技术阶段

- 目标：产出或更新技术文档
- 最低要求：改造点、数据模型、接口、时序、风控、测试重点
- 约束：技术文档必须基于已确认的需求文档
- 闸口：未获批准不得进入实现阶段

### 阶段 3：实现阶段

- 目标：按已确认文档完成代码改动
- 约束：只实现确认范围，不擅自扩需求
- 要求：实现完成后必须把实际落地结果回写到需求文档和技术文档
- 闸口：完成后停下，等待审核或测试反馈

### 阶段 4：评审吸收与修正阶段

- 目标：读取评审文档，吸收合理意见，修正代码与文档
- 约束：只读评审文档，不生成评审文档
- 要求：所有采纳的评审意见都要同步回写到需求文档和技术文档
- 闸口：每轮评审修正完成后都必须停下

## 文档回写最低要求

### 需求文档

- 当前需求结论
- 需求边界变化
- 已确认与未确认项
- 与评审相关的修正摘要
- 当前风险或上线判断

### 技术文档

- 当前技术方案
- 实际代码落地状态
- 被评审影响的接口、模型、流程、风控
- 每轮修复摘要
- 当前遗留问题与测试关注点

## 阶段输出模板

每个阶段结束时，统一按以下结构输出：

1. 本阶段完成内容
2. 产物路径
3. 文档是否已回写
4. 明确声明“本阶段已完成，等待审核确认”

## 执行检查清单

每次执行前必须自检：

- 当前所处阶段是否明确
- 是否已获得上一个阶段的用户确认
- 需求文档是否已更新
- 技术文档是否已更新
- 是否将评审文档视为只读输入
- 是否准备在阶段结束后停下

## 成功标准

一个成功执行本 Skill 的流程，应满足：

- 有完整的需求文档
- 有完整的技术文档
- 代码改动与文档一致
- 每一轮评审吸收过程都有记录
- 每个阶段之间都有明确的人审闸口
- 执行者只读取评审，不生成评审文档
